# `line_bento_parser_skill.md`

## 1. 技能概述 (Overview)
*   **技能名稱**：LINE 訂餐口語菜單結構化解析器 (Line Bento Parser Skill)
*   **主體任務**：接收 LINE 群組中管理員發送的口語化、非結構化主廚菜單文字，將其精準轉化為符合後端資料庫（Google Sheets `logs`）與 LINE Flex 訊息所預期的標準 JSON 結構[cite: 1]。

---

## 2. 核心解析邏輯與狀態機範例 (Core Logic & Parsing Rules)

大腦在推理此文字時，必須嚴格區分 **午餐 (LUNCH)** 與 **晚餐 (DINNER)** 兩套截聯邏輯[cite: 3]：

### ☀️ A軌：午餐解析邏輯 (LUNCH)
*   **定價原則**：午餐所有便當一律固定為 **$110** 元[cite: 3]。
*   **關鍵字過濾與截取機制**：
    先將內文以逗號、換行或空格切分，並剔除常見冗言贅字（如「今天有」、「今天」、「謝謝」）[cite: 3]：
    1.  **魚便當**：若區段包含「`魚便當`」，必須將「魚便當」字眼移除，剩餘內容作為 `{content}`，包裝為 `炸魚便當（{content}）`[cite: 3]。
    2.  **清蒸魚**：若區段包含「`清蒸魚是`」或「`清蒸魚`」，將其移除後剩餘內容作為 `{content}`，包裝為 `清蒸魚便當（{content}）`[cite: 3]。
    3.  **合菜**：若區段包含「`合菜的肉是`」或「`合菜`」，將其移除後剩餘內容作為 `{content}`，包裝為 `合菜便當（{content}）`[cite: 3]。

### 🌙 B軌：晚餐解析邏輯 (DINNER)
晚餐屬於自由單點制，大腦必須具備以下 5 種智慧狀態機[cite: 3]：

1.  **水餃群組交叉乘積**：
    *   *特徵*：`水餃有高麗菜、韭菜（生的110、熟的60）`[cite: 3]
    *   *規則*：口味與狀態價格必須交叉乘積。輸出如：`高麗菜水餃-生的($110)`、`高麗菜水餃-熟的($60)`[cite: 3]。
2.  **多尺寸斜線價格**：
    *   *特徵*：`酸辣湯25/50/100` 或 `貢丸湯30／50`[cite: 3]
    *   *規則*：2個價格預設為 `[小, 大]`，3個價格預設為 `[小, 中, 大]`[cite: 3]。
3.  **有/無 組合變化**：
    *   *特徵*：`北部粽有蛋90、無蛋85`[cite: 3]
    *   *規則*：獨立拆為兩項：`北部粽 有蛋($90)` 與 `北部粽 無蛋($85)`[cite: 3]。
4.  **「都是」同價結尾補全**：
    *   *特徵*：`香蕉、巧克力、草莓吐司都是50元`[cite: 3]
    *   *規則*：自動擷取最後一項的結尾詞（字數 $\le 3$），前面未結尾的品項需自動補齊字尾[cite: 3]。
5.  **常規單點與限量捕捉**：
    *   *特徵*：`芋頭可頌60` 或 `大雞腿飯120(限量5份)`[cite: 3]
    *   *規則*：捕捉品名與純數字價格。若內文包含「限量/只有/限 + 數字」，必須提取該整數並寫入 `limit` 欄位[cite: 3]。

## 3. 輸出 JSON Payload 精準規格 (Output JSON Schema)

```json
{
  "date": "YYYY-MM-DD",
  "meal": "LUNCH", 
  "vendor": "好媽媽便當",
  "deadlineAt": "YYYY-MM-DD HH:MM", 
  "note": "",
  "createdByUserId": "U...",
  "createdByName": "老闆",
  "items": [
    {
      "itemId": "T_LUNCH_20260717_0", 
      "name": "炸魚便當（午仔魚、秋刀魚）",
      "price": 110,
      "sort": 9000,
      "category": "動態解析"
    }
  ]
}
```

---

## 4. 異常處理與防呆 (Edge Cases)
*   若遇到完全無法識別價格或品名的行數，不可崩潰。應將該行名稱完整保留，價格設定為 `0`，並將 `category` 標記為 `"⚠️需確認"`[cite: 3]。

---

## 5. 系統整合實作參考
在實際開發時，建議參考 `main.py` 與 `hermes_worker.py` 的設計：
1.  **任務佇列 (Queue)**：利用 Google Firestore 的 `PENDING` 狀態作為任務儲存庫。
2.  **本地執行 (Local Execution)**：`hermes_worker.py` 不依賴 Webhook 的 5 秒時限，直接輪詢 Firestore 執行 LLM 耗時解析。
3.  **主動推播 (Proactive Push)**：解析完成後，利用 LINE Push API 將 Flex Message 主動送達群組，並引導用戶點擊私訊按鈕完成後續訂餐與結帳流程[cite: 2]。
